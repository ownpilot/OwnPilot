# Phase 11: Workspace Protocol Research

> Researched: 2026-04-05 | Branch: feature/v2-contextual-chat

## 1. Two Workspace Concepts

OwnPilot has **two distinct workspace systems** that serve different purposes:

### 1a. Sandbox Workspace (WorkspaceSelectorInfo)

- **Type definition:** `packages/ui/src/api/types/workspace.ts:1-16`
- **API:** `workspacesApi.list()` / `.create()` / `.delete()` in `packages/ui/src/api/endpoints/misc.ts:195-199`
- **Endpoint:** `/api/v1/workspaces` (CRUD)
- **Purpose:** Docker container-based isolated execution environments
- **Fields:** `id`, `name`, `description`, `status`, `containerStatus`, `storageUsage`
- **UI Component:** `WorkspaceSelector` in ChatPage header bar
- **Selection flow:** User picks from dropdown -> `onWorkspaceChange(id)` -> `useChatStore.setWorkspaceId(id)`
- **Storage:** Database-backed (server creates containers on demand)

### 1b. File Workspace (FileWorkspaceInfo / SessionWorkspaceInfo)

- **Type definition:** `packages/ui/src/api/types/workspace.ts:19-31`
- **API:** `fileWorkspacesApi.list()` / `.files()` / `.delete()` / `.cleanup()` in `misc.ts:270-290`
- **Endpoint:** `/api/v1/file-workspaces`
- **Purpose:** Lightweight session-based directories for AI-generated file operations
- **Fields:** `id`, `name`, `path`, `agentId`, `sessionId`, `description`, `tags`, `size`, `fileCount`
- **Backend implementation:** `packages/gateway/src/workspace/file-workspace.ts` (844 lines)
- **Storage:** Filesystem-based under `{dataDir}/workspace/{session-id}/`
- **Structure per session:** `scripts/`, `output/`, `temp/`, `downloads/`, `.meta.json`
- **Key function:** `getOrCreateSessionWorkspace(sessionId, agentId)` — lazy creation

**Critical difference:** File workspaces are auto-created per chat session and don't require user selection. Sandbox workspaces are explicitly selected by the user in the ChatPage UI.

## 2. WorkspaceSelector Component Lifecycle

**File:** `packages/ui/src/components/WorkspaceSelector.tsx`

### Mount Flow
1. Component mounts with `selectedWorkspaceId: string | null` prop
2. `useEffect` calls `fetchWorkspaces()` on mount
3. API call: `workspacesApi.list()` -> `GET /api/v1/workspaces`
4. If no workspace selected and list non-empty -> auto-selects first: `onWorkspaceChange(list[0].id)`

### User Interaction
- Click button -> dropdown menu with workspace list
- Click workspace -> `onWorkspaceChange(ws.id)` -> `setWorkspaceId` in ChatStore
- Create button -> modal -> `workspacesApi.create(name)` -> prepend to list
- Delete with confirmation dialog -> `apiClient.delete(/workspaces/${id})`
- Download as ZIP -> direct `fetch(/api/v1/workspaces/${id}/download)`
- Refresh button -> re-fetches list

### State
- Local state only (no persistence to localStorage)
- Parent (`ChatPage`) connects it to `useChatStore().workspaceId`

## 3. Chat Request Workspace Context Flow

### UI -> API (useChatStore.tsx)

**File:** `packages/ui/src/hooks/useChatStore.tsx:284-325`

The `sendMessage` function builds the fetch body:

```typescript
body: JSON.stringify({
  message: content,
  provider,
  model,
  stream: true,
  ...(currentSessionId && { conversationId: currentSessionId }),
  ...(agentId && { agentId }),
  ...(workspaceId && { workspaceId }),   // <-- workspace injected here
  ...(directTools?.length && { directTools }),
  // ...
})
```

Key observation: `workspaceId` is sent as a **top-level field** in the chat POST body alongside `provider`, `model`, `agentId`, etc.

### API -> Agent (chat.ts)

**File:** `packages/gateway/src/routes/chat.ts:312-332`

```typescript
// Workspace — set on every request (cheap), but prompt section only on first
const sessionId = body.workspaceId || body.conversationId || conversationId;
const sessionWorkspace = getOrCreateSessionWorkspace(sessionId, body.agentId);
agent.setWorkspaceDir(sessionWorkspace.path);

if (!isPromptInitialized) {
  const wsContext = getWorkspaceContext(sessionWorkspace.path);
  const workspaceInfo = `\n\n## File Operations\nWorkspace: \`${wsContext.workspaceDir}\`. Use relative paths for new files.`;
  agent.updateSystemPrompt(promptWithoutOldWs + workspaceInfo);
}
```

**Resolution priority for `sessionId`:**
1. `body.workspaceId` (explicit user selection from WorkspaceSelector)
2. `body.conversationId` (continuing existing conversation)
3. `conversationId` (auto-generated by agent engine)

**What happens:**
1. `getOrCreateSessionWorkspace()` creates a filesystem directory if it doesn't exist
2. `agent.setWorkspaceDir()` tells the agent engine where files go
3. On first message, system prompt gets a `## File Operations` section with workspace path
4. Subsequent messages in same conversation skip prompt injection (idempotent)

## 4. Sandbox Workspace vs Host-FS Workspace

### Sandbox Workspace (Current Implementation)
- Container-based isolation (Docker)
- Selected explicitly via `WorkspaceSelector` in ChatPage
- Has `containerStatus` field (`running`, `stopped`, etc.)
- Storage tracked via `storageUsage.usedBytes` / `storageUsage.fileCount`
- Downloaded as ZIP
- Used for code execution in isolated environments

### Host Filesystem Workspace (Documented but NOT Yet Implemented)
- **Documentation:** `HOST-FILESYSTEM-ACCESS.md` describes 3 security profiles
- **Env vars:** `OWNPILOT_HOST_FS`, `OWNPILOT_HOST_FS_LABEL`
- **Mechanism:** Docker bind mount (`-v /home/USER:/host-home:rw`)
- **Code status:** No references to `OWNPILOT_HOST_FS` found in gateway source code
- **The doc says:** Backend auto-registers a virtual file workspace pointing to host path
- **Reality:** This is a design document for future implementation

### File Workspace (Session-Based, Auto-Created)
- Lightweight directories under `{dataDir}/workspace/`
- Auto-created per chat session via `getOrCreateSessionWorkspace()`
- No Docker containers, just filesystem directories
- Listed via `fileWorkspacesApi.list()` on the UI's Files tab
- Path traversal protection via `validateWorkspaceId()` and resolved path checks

## 5. X-Project-Dir Header Flow (Bridge Context)

The `X-Project-Dir` header is **NOT part of the current workspace protocol**. It's a Bridge-specific concept used in the OpenClaw Bridge daemon for routing CC (Claude Code) spawns to specific project directories.

### Current Flow (No X-Project-Dir in OwnPilot)

OwnPilot uses a different mechanism:
1. UI sends `workspaceId` in POST body
2. Gateway resolves it to a filesystem path via `getOrCreateSessionWorkspace()`
3. Agent gets `setWorkspaceDir(path)` — this is the agent's working directory
4. System prompt gets `## File Operations\nWorkspace: \`/path/to/workspace\``

### Bridge Flow (Separate System)
- Bridge injects `X-Project-Dir` header when spawning CC instances
- This tells the bridge which host directory the CC should operate in
- OwnPilot gateway has `X-Runtime` header handling (for bridge provider routing) but no `X-Project-Dir`

### What Exists in Gateway for Bridge
**File:** `packages/gateway/src/routes/agent-cache.ts:243-255`
```typescript
if (localProv.name.startsWith('bridge-')) {
  headers['X-Runtime'] = localProv.name.replace('bridge-', '');
}
```

And in UI (`useChatStore.tsx:269-279`):
```typescript
const bridgeName = [provider, providerDisplayName].find(n => n.startsWith('bridge-'));
if (bridgeName) {
  chatHeaders['X-Runtime'] = bridgeName.replace('bridge-', '');
}
```

This is provider routing, not workspace/project directory context.

## 6. Proposed Approach: Sidebar Context -> Chat Injection

For the v2.0 Contextual Sidebar Chat feature, the key question is: **how should the current page/context be injected into chat messages?**

### Current Architecture Gaps

1. **WorkspaceSelector is ChatPage-only** — it lives in `ChatPage.tsx`, not in a global sidebar
2. **useChatStore is global** — available everywhere via React Context, already has `workspaceId` state
3. **No page context injection** — chat doesn't know what page the user is viewing
4. **No sidebar chat component** — there's no persistent chat panel in the sidebar/layout

### Recommended Injection Methods

#### Method A: System Prompt Context Injection (Server-Side)
- Extend the chat POST body with a `context` field: `{ page: string, entityId?: string, entityType?: string }`
- Gateway injects this into the system prompt alongside `## File Operations`
- Pro: Agent sees full context, can use tools appropriately
- Con: Requires gateway changes

#### Method B: Message Prefix Injection (Client-Side)
- Sidebar chat prepends context to user message: `[Context: WorkflowPage, workflow-id: abc123]\n{user message}`
- No gateway changes needed
- Pro: Simple, immediate
- Con: Visible in message, may confuse the model

#### Method C: Hybrid (Recommended)
- Add `pageContext` field to chat POST body (alongside existing `workspaceId`, `agentId`)
- Gateway builds a `## Current Context` section in system prompt on first message
- Include: page name, entity ID, entity type, related data summary
- Sidebar chat component reads current route via `useLocation()` and passes context automatically

### Data Flow for Method C

```
User on WorkflowsPage
  |
  v
SidebarChat component (new)
  |-- reads useLocation() -> /workflows/abc123
  |-- resolves entity: { type: 'workflow', id: 'abc123' }
  |-- calls useChatStore().sendMessage(content, { pageContext })
  |
  v
POST /api/v1/chat
  body: { message, provider, model, pageContext: { page: 'workflows', entityId: 'abc123', entityType: 'workflow' } }
  |
  v
Gateway chat.ts
  |-- On first message: inject ## Current Context section into system prompt
  |-- "User is viewing workflow 'abc123'. Workflow definition: {summary}"
  |-- Agent now has full context for contextual responses
```

| File | Change |
|------|--------|
| `packages/ui/src/components/SidebarChat.tsx` | **NEW** — persistent sidebar chat panel |
| `packages/ui/src/hooks/useChatStore.tsx` | Add `pageContext` to sendMessage params |

| `packages/gateway/src/routes/chat.ts` | Parse `pageContext`, inject into system prompt |
| `packages/gateway/src/middleware/validation.ts` | Add `pageContext` to chat schema |

### WorkspaceSelector Reuse Decision

The existing `WorkspaceSelector` is a **sandbox workspace selector** (Docker containers). For sidebar chat:
- **Don't reuse** WorkspaceSelector as-is — sidebar chat should auto-create file workspaces per session
- **Do reuse** `useChatStore.workspaceId` + `setWorkspaceId` — the plumbing is already there
- **Consider** letting sidebar chat share the same session workspace as the main ChatPage (via shared `sessionId`)
